## 📌 О проекте

Бинарная классификация банковских транзакций на **легитимные** и **мошеннические**. Главная сложность задачи — не сама модель, а **экстремальный дисбаланс классов**: на ~285 тысяч транзакций приходится всего 492 случая мошенничества. В таких условиях наивные подходы (accuracy, ROC-AUC, порог 0.5) систематически вводят в заблуждение, и основной фокус проекта — на **корректной постановке задачи**, а не на «накручивании» метрик.

Проект доведён до рабочего демо: обученная модель упакована и развёрнута в виде интерактивного приложения, где можно подвигать порог классификации и увидеть, как меняется решение.

## 🎯 Почему PR-AUC, а не accuracy и не ROC-AUC

Это ключевое решение всего проекта. При дисбалансе 1:578:

- **Accuracy бесполезна.** Модель-пустышка, всегда предсказывающая «не мошенник», даёт **99.83% accuracy** — и при этом не ловит ни одного мошенника. Метрика выглядит отлично, ценность — нулевая.
- **ROC-AUC вводит в заблуждение.** ROC-AUC оперирует долей ложных срабатываний (FPR), а при огромном числе легитимных транзакций FPR остаётся низким даже при большом абсолютном количестве ложных тревог. Метрика выглядит оптимистично там, где модель на практике захлёбывается.
- **PR-AUC честнее.** Precision-Recall AUC напрямую связывает **полноту поимки мошенников** (recall) с **долей ложных тревог среди срабатываний** (precision) — именно то, что важно бизнесу. Baseline для PR-AUC равен доле позитивного класса (≈0.0017), поэтому любой реальный прогресс сразу виден.

**Поэтому все модели в проекте оптимизировались и сравнивались по PR-AUC.**

## 🧪 Подход

1. **Валидация целостности данных** — `assert`-проверки (отсутствие пропусков, корректность типов и баланса классов) выполняются **до** обучения, чтобы исключить «тихие» баги в пайплайне.
2. **Feature engineering** — лог-преобразование `Amount` (`log1p`) для сжатия тяжёлого правого хвоста распределения сумм.
3. **Модель** — `LightGBM` как сильный и быстрый baseline для табличных данных.
4. **Подбор гиперпараметров** — `Optuna`, **40 trials**, оптимизация по **PR-AUC** на **стратифицированной 5-fold кросс-валидации** (стратификация обязательна при таком дисбалансе).
5. **Анализ порога** — cost-sensitive подбор порога классификации вместо фиксированных 0.5.

Все эксперименты ставились **чисто — по одному фактору за раз**, чтобы каждый прирост метрики был объясним.

## 📈 Результаты

Путь улучшений на отложенном тесте (метрика — **PR-AUC**):

| Этап | PR-AUC | Δ |
|---|---:|---|
| Baseline (`DummyClassifier`) | 0.0017 | — |
| LightGBM (дефолтные параметры) | 0.3534 | +0.3517 |
| \+ Feature engineering (`log Amount`) | 0.3683 | +0.0149 |
| \+ Optuna (40 trials, 5-fold CV) | **0.876** | **+0.508** |

> Итоговая модель достигает **PR-AUC = 0.876** на отложенном тесте — при baseline-уровне 0.0017.

### Заметки по экспериментам

- **`Time` исключён из признаков.** Проверено экспериментально: оттюненная модель работает **лучше без него**. Решение принято по результату, а не по интуиции.
- **Оптимальный порог зависит от стоимости ошибок.** Пропустить мошенника дорого (теряется сумма транзакции), ложно заблокировать честного — дёшево (звонок в поддержку). Поэтому cost-sensitive анализ показывает, что **оптимум обычно лежит ниже 0.5**, а единственно «правильного» порога не существует — он диктуется бизнесом.

## 🚀 Демо

> **Живое демо:** **[credit-card-fraud-detection-sm1ley.streamlit.app](https://credit-card-fraud-detection-sm1ley.streamlit.app/)**

Интерактивное приложение на **Streamlit**: выбираешь тип транзакции (мошенническая / честная / сомнительная), двигаешь порог классификации и наблюдаешь, как меняется решение модели. Особенно показательны «сомнительные» примеры (вероятность 0.2–0.8) — на них видно, зачем вообще нужен настраиваемый порог.

```bash
streamlit run app.py
```

## ⚙️ Установка и запуск

```bash

```

## 🔑 Что выделяет проект

- ✅ **Осознанный выбор метрики под задачу** — PR-AUC вместо accuracy/ROC-AUC, с объяснением почему.
- ✅ **Валидация целостности данных** через `assert` до обучения.
- ✅ **Чистый эксперимент** — изменения по одному фактору за раз, каждый прирост объясним.
- ✅ **Стратифицированная CV** при подборе гиперпараметров — корректно для дисбаланса.
- ✅ **Cost-sensitive анализ порога** с выводом, что оптимум определяется ценой ошибок, а не дефолтным 0.5.
- ✅ **Доведено до продукта** — обученная модель развёрнута в интерактивном демо.
